Post-M44: Linked Source Test Workflow
Post-M44 #2347
M44 delivers the foundation. The source-level opt-in setting, collection linked source config, and resolution behavior all ship in M44. The guided workflow UI — the "Try New Version" wizard — is post-M44.
What M44 delivers
  • Authorization gate: user must have HEAD access to the source (existing OCL source-level permissions — no separate opt-in flag needed)
  • Collection → Settings → Linked Sources table with HEAD-link flag per source relationship
  • Resolution: when HEAD-link flag is set, unversioned refs resolve against HEAD of that source while working in collection HEAD
  • UI: "Linked Source" chip in collection header
What Post-M44 adds
  • Guided "Try New Version" wizard (replaces manual Create Similar)
  • Impact preview before committing
  • "Commit" or "Cancel" the upgrade in one guided flow
  • Transition path: linked source → locked repo version at release time
M44 → Post-M44 Progression for Step 3
M44
User manually uses "Create Similar" to test a new source version. Works — but requires knowing the feature exists and navigating to the right panel.
Post-M44
Guided "Try New Version" button in the update workflow. Auto-creates the test expansion, shows impact, and offers "Commit" or "Cancel" in one flow. No detour required.
ADR-007 (updated 2026-05-13): HEAD-resolution is not limited to same-namespace sources. Authorization is permission-based — any user with HEAD access to a source may HEAD-link it from their collection, regardless of org. Cross-org canonical URL resolution is handled by the existing URL registry.

Open Questions for Discussion
Discussion
1. Step 2 vs Step 5 diff granularity
Step 2 shows CIEL-version-level changes. Step 5 shows collection-expansion changes. Is this the right split? Should Step 2 already be scoped to concepts in this collection only?
2. Retired concepts — what's the default?
When CIEL retires a concept that's referenced in the collection, does the expansion keep it (marked Retired) or remove it? Needs input from Andy (governance) + Burke (OpenMRS export impact).
3. Reference type filter chips (#2502)
Some filter chips on the References tab partially work. Before fixing: what's the canonical chip list, and what does each one filter on? Needs a spec before we can validate the bug fix.
4. Is "Create Similar" enough for Step 3?
For the M44 showcase, "Create Similar" is the test path. Is it discoverable enough, or do we need a thin wrapper (e.g., "Test with new version" button) that pre-fills the form?
5. Collection banner — dismissibility
The yellow banner notifying of a pending dependency update — permanently dismissible? Session-only? Or only cleared by completing the update workflow?
