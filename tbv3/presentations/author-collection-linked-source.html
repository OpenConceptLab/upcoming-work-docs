Authoring a Collection with a Linked (HEAD) Source

How the HEAD-link flag lets implementers see new source content in their collection immediately during authoring — and transitions cleanly to a released version on publish.
Jump to The Problem The HEAD-Link Phase 1: Authoring Phase 2: Releasing Unreleased Warning Design Decisions Open Issues
1 Context & Problem

Why does this exist?

A Terminology Implementer owns both a source and a collection — for example, the PIH source and the PIH dictionary collection. They add new concepts to the source and add references to those concepts in the collection at the same time.

The problem: New concepts added to a source's HEAD are invisible in the collection until the source is versioned and released. This breaks the authoring loop — you can't see your work.

Without HEAD-link: Create concept in source → must release source → only then visible in collection. Each round-trip costs a full release cycle.
With HEAD-link: Create concept in source → immediately visible in collection HEAD. Release only when you're ready to publish.
Notes
2 Core Concept

What is the HEAD-link flag?

A HEAD-link is a flag on the collection's relationship to a specific source. It says:

"While I am working in HEAD of this collection, resolve references to this source against HEAD of that source."

This is not a new entity. It is a property of the existing collection-to-source linkage. It uses the existing resolve reference operation — just with an additional mode for authoring.

Design decisions at a glance

Decision Choice Rationale
Where is the flag configured? Collection level per source relationship The collection controls how it resolves; the source doesn't know or care.
Authorization basis Permission to access HEAD of the source Resolution must be consistent across all users of the same collection. Namespace is the most common way permission is granted, but it is not the gate — users with HEAD access to a source in another org can use the HEAD-link. The URL registry handles cross-org canonical URL resolution.
Does the reference text change? No The reference expression stays the same; only resolution behavior differs.
What other sources are affected? Only the HEAD-linked source External sources (e.g., CIEL) stay pinned to their locked released version.

3 Phase 1

Authoring — working in HEAD

The user adds new content to their source and collection simultaneously. The HEAD-link ensures that new source content is immediately visible in the collection without requiring a release cycle.

AUTHORING PHASE — working in HEAD of both source and collection ═══════════════════════════════════════════════════════════════════ PIH Collection [HEAD] ┌──────────────────────────────────────────────────────────────┐ │ ref: /PIH/sources/PIH/concepts/123/ ─────────────────┐ │ │ ref: /PIH/sources/PIH/concepts/124/ ─────────────────┤ │ │ ref: /CIEL/sources/CIEL/concepts/5089/ ───────────────┐ │ │ └──────────────────────────────────────────────────────────────┘ │ HEAD-link flag active │ │ ▼ │ │ PIH Source [HEAD] ◄────────────────────────────────────────┘ ┌────────────────────────┐ │ concept/123 ← new! │ resolved immediately ✓ │ concept/124 ← new! │ user sees these in collection └────────────────────────┘ │ ▼ CIEL [v2026-01] ← locked, unchanged ┌──────────────────────┐ │ concept/5089 │ └──────────────────────┘

What the user experiences

  1. User creates a new concept in PIH Source → concept lands in PIH Source HEAD.
  2. System adds a reference to the collection HEAD (or user adds it manually).
  3. Auto-expansion resolves the reference against PIH Source HEAD via the HEAD-link.
  4. User navigates to the collection → sees the new concept immediately.
  5. Repeat for each new concept — no release required to see the work.
Under the hood: Resolve reference is called with a flag to use HEAD of the linked source. The reference expression (e.g., /PIH/sources/PIH/concepts/123/) is unversioned — the same expression will appear in the released version. CIEL is unaffected.

4 Phase 2

Releasing — batch release of source then collection

When the user is ready to publish, two things must happen in order:

  1. Release the source — unpublished HEAD content must be versioned first.
  2. Release the collection — the HEAD-link switches from HEAD to the newly released source version.
RELEASE PHASE — batch release of source then collection ═══════════════════════════════════════════════════════════════════ STEP 1: Release PIH Source PIH Source [HEAD] ┌────────────────────┐ │ concept/123 │ ──publish──► PIH Source [v19.3] │ concept/124 │ ┌────────────────────┐ └────────────────────┘ │ concept/123 ✓ │ │ concept/124 ✓ │ └────────────────────┘ STEP 2: Release Collection PIH Collection [HEAD] ──publish──► PIH Collection [v3.0] The HEAD-link switches on release: ┌──────────────────────────────────────────────────────────────┐ │ BEFORE (HEAD): resolve PIH refs → PIH Source HEAD │ │ AFTER (v3.0): resolve PIH refs → PIH Source v19.3 ✓ │ └──────────────────────────────────────────────────────────────┘ RESULT: PIH Collection v3.0 ┌──────────────────────────────────────────────────────────────┐ │ ref: /PIH/sources/PIH/concepts/123/ → PIH v19.3 ✓ │ │ ref: /PIH/sources/PIH/concepts/124/ → PIH v19.3 ✓ │ │ ref: /CIEL/sources/CIEL/concepts/5089/ → CIEL v2026-01 ✓ │ └──────────────────────────────────────────────────────────────┘ All references resolve. HEAD-link is no longer active in v3.0.

Happy path steps

  1. User decides to publish a new version of the dictionary.
  2. System checks whether the HEAD-linked source has any unreleased content referenced by the collection.
  3. If yes: user is prompted to release PIH Source first (see warning below).
  4. User releases PIH Source (e.g., as v19.3).
  5. User releases the collection — system re-evaluates references and locks to PIH Source v19.3.
  6. Collection version is published. All references resolve cleanly.
Notes
5 Edge Case

Unreleased content warning

If the user tries to release the collection before releasing the source, the system surfaces a warning — not a hard block. The user can proceed, but references pointing to unreleased source content will return no result in the published collection version.

⚠️ Heads up (sample UI message): Your collection references content in PIH Source that has not been published yet (N concepts in HEAD). If you release the collection now, those references will not resolve in the published version.

Recommended: Release PIH Source first, then return to release the collection.

Release PIH Source first →  Continue anyway

6 Reference

What stays locked during authoring

HEAD-link is opt-in, per source, at the collection level. Only the source explicitly configured as HEAD-linked gets the HEAD resolution behavior. All other sources stay pinned.

Source Relationship type During authoring HEAD On release
PIH Source (own source) HEAD-linked Resolves to PIH Source HEAD Transitions to latest released PIH Source version
CIEL Locked to v2026-01 Resolves to CIEL v2026-01 (unchanged) Stays at v2026-01 unless explicitly updated

What already exists vs. what needs to be built

Status Item
Exists Resolve reference supports a HEAD parameter and falls back to HEAD when no release exists
Exists Source version locking on collection release
Exists Namespace-based URL registry for multi-org resolution
Needs build UI to configure a source relationship as HEAD-linked (collection settings)
Needs build Business logic in auto-expansion to pass the HEAD flag for HEAD-linked sources
Needs build Re-evaluation on collection release (currently copies HEAD auto-expansion; must re-evaluate instead)

7 Open Issues

Known open questions

Issue Description
Auto-expansion re-evaluation on release OCL currently copies the HEAD auto-expansion when creating a released version. For the HEAD-link transition to work, the release step must re-evaluate references and lock to the released source version. This is a behavior change in the current release flow.
Dirty reference detection If a concept in the source HEAD is edited or deleted, the collection HEAD does not automatically refresh. Re-running all references on every source change is too slow at scale. MVP may require manual refresh; a smarter invalidation mechanism (e.g., source-level change timestamps vs. expansion timestamps) is needed longer-term.
Multi-org namespaces Resolved Authorization is permission-based, not namespace-based — a user with HEAD access to a source in another org (e.g., MSF OCB) can HEAD-link it from their collection. The URL registry already handles cross-org canonical URL resolution. Namespace is the common case, not the gate. (Decided 2026-05-13 squad call.)
