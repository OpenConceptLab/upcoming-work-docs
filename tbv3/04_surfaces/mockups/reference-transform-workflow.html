Ticket #2433 with #2274 context

Bulk transform collection references before implementation starts

This walkthrough shows how an editor selects references, groups mixed reference types, previews before/after expressions, handles very large references, and commits only the rows that are ready. It is meant to help product, engineering, and leadership validate the shape of the workflow before oclweb3 development begins.

What users are trying to do

Clean up old reference expressions, unpin versioned references, or lock HEAD-resolving references to a released repo version.

What makes this tricky

A single selection can contain unversioned, repo-versioned, deprecated resource-versioned, intensional, and unsupported references.

What the UI must protect

Users need to see what will change, which rows are skipped, and when a reference is too large for immediate resolution preview.

Core principle: a transform can be structurally valid even when OCL cannot synchronously resolve all referenced resources. The UI should still show the before/after expression and make the resolution uncertainty explicit.
End-to-end workflow

One bulk action, with per-group and per-row safety

1 Select references

User filters the References tab and checks one or more rows. The toolbar shows selected count and enables Transform on HEAD.

2 Classify selection

The dialog groups references by pinning/expression pattern so each group gets appropriate transform choices.

3 Configure groups

User chooses a transform for one or more groups. Unsupported groups remain visible but skipped.

4 Preview changes

Each row shows current and proposed expressions, status, and resolution preview when OCL can return it quickly.

5 Commit ready rows

Only Ready rows are submitted. The result summary reports transformed, skipped, failed, and queued re-expansion.

For developers

The dialog should model selection as groups first, rows second. Each row has independent eligibility, preview, and commit state.

For directors

The workflow favors transparency over magic: users see what changes, what does not, and which outcomes are not fully knowable until re-expansion.

Structural preview

Checks whether the current expression can be parsed, whether the transform is allowed, and what the proposed expression would be. This should be fast and available for every supported row.

Resolution preview

Checks resolved source/version and resource counts when OCL can evaluate the reference quickly. If not, the row uses Preview pending or Preview too large rather than failing the whole dialog.

Validation checklist

Decisions to make before development

API shape

Do we have one bulk transform preview/commit endpoint, or does the client orchestrate lower-level operations?

Async preview

What API statuses and thresholds identify references that are too large to resolve synchronously?

Commit without full resolution

Can users commit structurally valid rows when resolved counts are unavailable, and should that require extra confirmation?

Version defaults

When locking to repo version, should the default be the expansion-locked version, latest released version, or explicit user choice?

Resource-version mapping

How should OCL pick a repo version when a deprecated resource-versioned URL maps to multiple released repo versions?

Reference identity

Does a transform preserve the reference ID, or delete and recreate the reference with a new ID?

Proposed MVP stance: support grouped transforms for Unpin, Lock, and deprecated-resource conversion; include expression rewrite only if backend validation can make it safe enough for bulk use.
